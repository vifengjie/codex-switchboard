# DOC-001 文档体系与命名规范

文档编号：DOC-001  
状态：草案  
最后更新：2026-05-03

## 1. 目标

建立可长期继承的产品文档体系，使产品背景、需求分析、产品设计、技术方案、开发任务拆解、验证记录能够独立维护、互相引用、持续演进。

## 2. 目录规范

| 目录 | 说明 |
|---|---|
| `00-governance/` | 文档体系、命名规范、决策记录 |
| `01-background/` | 产品背景、用户画像、业务约束、合规边界 |
| `02-requirements/` | 需求分析、需求池、验收标准 |
| `03-product-design/` | 产品设计、信息架构、交互、页面状态 |
| `04-technical-solution/` | 技术架构、数据模型、采集方案、账号切换方案 |
| `05-development-plan/` | 版本计划、任务拆解、排期、发布清单 |
| `06-validation/` | PoC、实验记录、待验证问题、测试结论 |
| `90-assets/` | 图、截图、原型、导出素材 |

## 3. 文件命名规范

文件名采用：

```text
<TYPE>-<NNN>-<short-english-slug>.md
```

示例：

```text
REQ-001-codex-multi-account-quota-management.md
TECH-001-architecture-and-implementation-options.md
DEV-001-mvp-task-breakdown.md
```

编号类型：

| 类型 | 含义 |
|---|---|
| `DOC` | 文档治理 |
| `BKG` | 产品背景 |
| `REQ` | 需求分析 |
| `PRD` | 产品设计 |
| `TECH` | 技术方案 |
| `DEV` | 开发计划 |
| `VAL` | 验证计划 |
| `ADR` | 架构或产品决策记录，后续需要时新增 |

## 4. 文档头规范

每个正式文档建议包含：

```text
文档编号：
文档状态：
负责人：
最后更新：
关联文档：
```

文档状态建议使用：

| 状态 | 含义 |
|---|---|
| 草案 | 正在形成，不作为研发承诺 |
| 评审中 | 可供讨论，需要确认 |
| 已确认 | 当前版本可作为实现依据 |
| 已归档 | 被新文档替代，仅供追溯 |

## 5. 继承规则

- `BKG` 解释为什么做。
- `REQ` 定义做什么、做到什么程度。
- `PRD` 定义用户如何使用。
- `TECH` 定义如何实现。
- `DEV` 定义如何拆解交付。
- `VAL` 记录哪些假设被验证、哪些仍有风险。

原则上不要在一个文档里长期混写所有阶段内容。早期可以在 `REQ-001` 中承载较多分析，但进入产品设计和开发后，应逐步拆到对应目录。

## 6. 图与素材规则

- 流程图、架构图优先使用 Mermaid，便于版本维护。
- 截图、原型图、AI 生成图片放在 `90-assets/images/`。
- Mermaid 源文件或复杂图说明放在 `90-assets/diagrams/`。
- 文档引用图片时使用相对路径，例如 `../90-assets/images/menu-bar-demo.png`。
